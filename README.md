# Интернет-магазин «Web-Larёk» 
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер. 
 
## Архитектура приложения 
 
Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность: 
 
Model - слой данных, отвечает за хранение и изменение данных.   
View - слой представления, отвечает за отображение данных на странице.   
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных. 
 
Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений. 
 
### Базовый код 
 
#### Класс Component 
Является базовым классом для всех компонентов интерфейса. 
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения. 
 
Конструктор:   
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает. 
 
Поля класса:   
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента. 
 
Методы класса:   
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.   
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>` 
 
 
#### Класс Api 
Содержит в себе базовую логику отправки запросов. 
 
Конструктор:   
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов. 
 
Поля класса:   
`baseUrl: string` - базовый адрес сервера   
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов. 
 
Методы:   
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер   
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.   
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных. 
 
#### Класс EventEmitter 
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления. 
 
Конструктор класса не принимает параметров. 
 
Поля класса:   
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события. 
 
Методы класса:   
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.   
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.   
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра. 
 
 
## Данные 
 
В приложении используются две основные сущности данных: товар и покупатель. 
 
```ts 
interface IProduct { 
  id: string; 
  description: string; 
  image: string; 
  title: string; 
  category: string; 
  price: number | null; 
} 
``` 
 
Интерфейс IProduct описывает структуру данных товара, которые используются 
для отображения каталога товаров, карточки товара и работы с корзиной. 
 
```ts 
type TPayment = 'card' | 'cash'; 
 
interface IBuyer { 
  payment: TPayment; 
  email: string; 
  phone: string; 
  address: string; 
} 
``` 
 
Интерфейс IBuyer описывает данные покупателя, которые вводятся пользователем 
при оформлении заказа. 
 
 
## Модели данных 
 
### Каталог товаров 
 
Класс отвечает за хранение и управление товарами, доступными для покупки в приложении. 
 
Поля: 
- products: IProduct[] — массив всех товаров, полученных с сервера; 
- selectedProduct: IProduct | null — товар, выбранный для подробного отображения. 
 
Методы: 
- setProducts(products: IProduct[]) — сохраняет массив товаров; 
- getProducts(): IProduct[] — возвращает массив товаров; 
- getProductById(id: string): IProduct | undefined — возвращает товар по его идентификатору; 
- setSelectedProduct(product: IProduct) — сохраняет выбранный товар; 
- getSelectedProduct(): IProduct | null — возвращает выбранный товар. 
 
 
### Корзина 
 
Класс отвечает за хранение товаров, выбранных пользователем для покупки. 
 
Поля: 
- items: IProduct[] — массив товаров, находящихся в корзине. 
 
Методы: 
- getItems(): IProduct[] — возвращает список товаров в корзине; 
- addItem(product: IProduct) — добавляет товар в корзину; 
- removeItem(product: IProduct) — удаляет товар из корзины; 
- clear() — очищает корзину; 
- getTotalPrice(): number — возвращает общую стоимость товаров в корзине; 
- getItemsCount(): number — возвращает количество товаров в корзине; 
- hasItem(id: string): boolean — проверяет наличие товара в корзине по его id. 
 
 
### Покупатель 
 
Класс отвечает за хранение данных покупателя и их валидацию при оформлении заказа. 
 
Поля: 
- payment: TPayment | null — выбранный способ оплаты; 
- address: string — адрес доставки; 
- email: string — email покупателя; 
- phone: string — телефон покупателя. 
 
Методы: 
- setData(data: Partial<IBuyer>) — сохраняет данные покупателя, не затирая уже существующие; 
- getData(): IBuyer — возвращает данные покупателя; 
- clear() — очищает данные покупателя; 
- validate(): Record<string, string> — выполняет валидацию данных покупателя и возвращает объект ошибок. 
 
Пример результата валидации: 
 
```ts 
{ 
  payment: 'Не выбран вид оплаты', 
  email: 'Укажите email' 
} 
``` 
 
 
## Слой коммуникации 
 
Слой коммуникации отвечает за взаимодействие приложения с сервером «Веб-ларёк». 
Он предназначен для получения данных о товарах и отправки данных заказа на сервер. 
 
Классы данного слоя не хранят состояние приложения, не работают с DOM и не 
содержат бизнес-логики. Их задача — выполнять HTTP-запросы и возвращать 
результаты в удобном для использования виде. 
 
### Класс WebLarekApi 
 
Класс реализует работу с API сервера и использует композицию — принимает 
экземпляр базового класса Api из стартового набора. 
 
#### Назначение класса 
- получение каталога товаров с сервера; 
- отправка данных заказа на сервер. 
 
#### Конструктор 
- constructor(api: IApi) 
  - api: IApi — экземпляр класса Api, предоставляющий методы get и post 
    для выполнения HTTP-запросов. 
 
#### Методы класса 
- getProducts(): Promise<IProduct[]> 
  - выполняет GET-запрос на эндпоинт /product/; 
  - возвращает массив товаров, полученных с сервера. 
 
- postOrder(order: IOrder): Promise<IOrderResponse>
  - IProductsResponse — объект ответа сервера на запрос каталога товаров.
  - IOrderResponse — объект ответа сервера после оформления заказа.
  - выполняет POST-запрос на эндпоинт /order/; 
  - принимает объект заказа, содержащий данные покупателя и список товаров; 
  - отправляет данные заказа на сервер и возвращает ответ сервера. 


# Слой View

Слой View отвечает за отображение данных на странице и генерацию событий при взаимодействии пользователя с интерфейсом.

Компоненты представления:
- не хранят состояние приложения;
- не содержат бизнес-логики;
- не принимают решений;
- получают данные через метод render;
- уведомляют приложение о действиях пользователя через события.

Все компоненты представления наследуются от базового класса Component.

Все DOM-элементы, с которыми взаимодействует компонент, находятся в конструкторе класса и сохраняются в его полях. 
Слушатели событий устанавливаются один раз в конструкторе. 
Компоненты не хранят данные приложения и не содержат бизнес-логики.

# События в приложении


## События моделей

### products:changed
Генерируется в методе setProducts(products: IProduct[]) при сохранении массива товаров, полученных с сервера.
Используется для перерисовки каталога товаров.

### product:selected
Генерируется в методе setSelectedProduct(product: IProduct) при выборе товара для подробного просмотра.
Используется для открытия модального окна с карточкой товара.

### basket:changed
Генерируется при любом изменении содержимого корзины:
- в методе addItem(product: IProduct) — при добавлении товара;
- в методе removeItem(product: IProduct) — при удалении товара;
- в методе clear() — при очистке корзины.

Используется для:
- обновления списка товаров в корзине;
- пересчёта общей стоимости;
- обновления счётчика товаров в Header;
- активации или деактивации кнопки «Оформить».

### buyer:changed
Генерируется:
- в методе setData(data: Partial<IBuyer>) — при изменении данных покупателя;
- в методе clear() — при очистке данных после успешного оформления заказа.

Используется для:
- повторной валидации форм;
- обновления состояния кнопок отправки;
- очистки форм после завершения заказа.


## События представлений:

- card:select
- card:add
- card:remove
- basket:remove
- basket:open
- order:start
- order:next
- order:submit
- form:change

# Презентер

Презентер отвечает за связь между слоями Model и View.

Презентер реализован в файле main.ts. 
Так как приложение является одностраничным, используется единый презентер. 
Логика приложения построена на подписке на события моделей и представлений.

Функции презентера:
- подписка на события моделей;
- подписка на события представлений;
- вызов методов моделей;
- обновление представлений при изменении данных;
- управление открытием и закрытием модальных окон.

Презентер не генерирует события и не хранит данные.

Перерисовка представлений происходит только:
- при обработке событий моделей, сообщающих об изменении данных;
- либо при открытии модального окна.

Презентер обрабатывает события, но не инициирует их.
